# ClipCascade 统一需求与技术规格

更新时间：2026-04-04

本文档是当前唯一有效的需求文档，也是实现、测试、README、后续发布时必须对齐的真源。

## 1. 产品边界

ClipCascade 只做一件事：

让一台设备复制出来的内容，尽量像“本机剪贴板”一样，在另一台设备上继续用。

当前强保证范围：

- `server`
- Windows 无 UI 桌面端
- macOS 无 UI 桌面端
- Linux 无 UI 桌面端

当前支持的内容类型：

- 文本
- 图片
- 单个图片文件
- 单个普通文件
- 多个普通文件
- 单个文件夹
- 多个文件夹

当前不承诺的重点：

- 移动端文件链路的强保证交付
- 把文件直接后台复制进 Finder / Explorer / Dolphin 当前目录
- 持久化到数据库的长期历史

### 1.1 无桌面 Linux 模式的单独定位

这里的“无桌面 Linux”指：

- 没有 X11
- 没有 Wayland
- 没有系统托盘
- 没有统一图形系统剪贴板
- 典型运行场景是 `ssh`、纯 TTY、机房物理终端、服务器控制台

这类环境不能继续套用桌面版的这些能力：

- 全局热键注册
- 系统图形剪贴板读写
- 文件型剪贴板
- 自动模拟 `Ctrl+V`

因此，无桌面 Linux 必须作为一套单独模式来设计。
它的核心目标不是“控制图形剪贴板”，而是：

- 让共享内容能进入当前 shell / 终端工作流
- 让文本能直接进入命令行
- 让图片和文件能落盘后把路径快速带到当前命令行

这套模式当前仅作为需求保留，尚未实现。

## 2. 两个关键概念

### 2.1 共享历史

ClipCascade 维护的是“全终端共享剪贴板历史”。

- 发送端复制后，内容会进入这份历史
- 接收端收到广播后，内容也会进入这份历史
- Web 控制中心显示的就是这份历史

### 2.2 `ReservedPaths` 和 `LocalPaths`

大白话：

- `ReservedPaths`：预定坑位。意思是“将来准备落到哪里”
- `LocalPaths`：已经到货。意思是“真实文件已经在这台机器上了”

文件或图片还没真正落盘时，可以先只有 `ReservedPaths`。
真正落盘成功以后，`LocalPaths` 才会写进去。

## 3. 默认快捷键

说明：

- `Ctrl+C` / `Cmd+C` 不是 ClipCascade 主动拦截的发送热键，而是系统原生复制；ClipCascade 监听复制结果并自动广播
- `Ctrl+Alt+Shift+C` 是手动补发当前系统剪贴板

| 功能 | Windows / Linux | macOS | 实际行为 |
| --- | --- | --- | --- |
| 原生复制并自动广播 | `Ctrl+C` | `Cmd+C` | 系统先复制，ClipCascade 再监听并广播 |
| 手动补发当前剪贴板 | `Ctrl+Alt+Shift+C` | `Cmd+Option+Shift+C` | 强制再发一次当前系统剪贴板 |
| 图片路径粘贴 | `Ctrl+Alt+V` | `Cmd+Option+V` | 只对图片类有效，把 tmp 路径写入系统剪贴板，并尽量自动模拟一次粘贴 |
| 真实内容写入剪贴板 | `Ctrl+Alt+Shift+V` | `Cmd+Option+Shift+V` | 对图片和文件类有效，只把真实结果写入系统剪贴板，不模拟粘贴 |

## 4. 按内容类型的实际逻辑

### 4.1 文本

发送端：

- 用户按系统原生 `Ctrl+C` / `Cmd+C`
- 文本立即广播给同账号下全部在线客户端
- 发送端自己只记录历史，不再二次改写本机系统剪贴板

异机接收端：

- 收到文本后立即写入本机系统剪贴板
- 用户直接按系统原生 `Ctrl+V` / `Cmd+V` 即可

### 4.2 图片、截图、单个图片文件

发送端：

- 用户按系统原生复制
- 图片内容立即广播
- 单个图片文件走图片即时链路，不走普通 `file_stub`

异机接收端：

- 先把图片内容收进 ClipCascade 内存
- 再把图片内容写入本机系统剪贴板
- 所以用户可以直接原生 `Ctrl+V` / `Cmd+V` 粘贴图片本身

`Ctrl+Alt+V`：

- 只对图片类有效
- 先规划一个 tmp 路径，例如 `.../ClipCascade/1.jpg`
- 立刻把这个路径文本写入系统剪贴板
- 尝试自动模拟一次粘贴
- 如果真实图片文件还没落盘，就在后台继续把内存中的图片物化到同一路径

`Ctrl+Alt+Shift+V`：

- 若图片还未物化，则先物化到 tmp
- 然后把真实图片文件写入系统文件型剪贴板
- 然后尽量自动模拟一次系统原生 `Ctrl+V` / `Cmd+V`

### 4.3 普通文件、多个文件、文件夹

发送端：

- `Ctrl+C` 时只广播文件描述，不立即传真实文件内容
- 主链路协议是：
  `file_stub -> file_request -> file_chunk -> file_complete -> file_release`

接收端按 `Ctrl+Alt+V`：

- 对非图片文件类无效
- 当前设计就是“没反应”

接收端按 `Ctrl+Alt+Shift+V`：

- 如果文件还没传输，就发起或复用传输
- 如果文件正在传输，就保持下载中状态，不重复开新传输
- 如果文件已经在 tmp 里准备好了，就直接复用现有结果
- 传输完成后，把真实文件或文件列表写入系统文件型剪贴板
- ClipCascade 不再模拟一次系统粘贴
- 用户自己到目标目录按系统原生 `Ctrl+V` / `Cmd+V`

这条热键的重点是：

- 只传一次
- 只落一次
- 后续重复按热键只复用已有 tmp 结果

### 4.4 自发自收

自发自收指发送端和接收端其实是同一个客户端实例。

统一规则：

- 收到自己的广播后，仍然要进入共享历史
- Web 控制中心也要能看到
- 但文本和图片收到自己广播时，不再二次改写本机系统剪贴板

图片和文件在自发自收场景下：

- 仍然保留 `Ctrl+Alt+V` / `Ctrl+Alt+Shift+V`
- 不直接复用发送端当时的原始源路径
- 仍然走“接收端 tmp 目录准备逻辑”
- 这样异机和自发自收体验统一，用户不用学两套规则

## 5. tmp 目录与命名规则

统一 tmp 根目录：

- `os.TempDir()/ClipCascade`

命名规则：

- 单个图片：优先保留发送端原始文件名
- 截图或没有文件名的图片：时间戳 `*.png`
- 单个普通文件：保留原文件名
- 单个文件夹：保留原文件夹名
- 多文件 / 多文件夹 / 混合选择：使用时间戳目录

目录结构规则：

- 多文件在 tmp 中必须保留原目录结构和原文件名
- 单个文件夹传输后，tmp 中保留同名目录

覆盖规则：

- 同名同内容：直接复用
- 同名不同内容：覆盖旧内容
- 不自动生成 `_1`、`_2` 之类副本

安全规则：

- 所有自动规划和落盘路径都必须限制在 `ClipCascade` 的 tmp 根下
- 不允许路径穿越
- 不允许借历史记录或 zip 条目把文件写到 tmp 根外面

## 6. 幂等性要求

幂等性是当前版本的硬要求。

必须满足：

- 同一份未变化的源文件重复复制，不生成新的传输 ID
- 同一个 `file_request` 重复触发，不重复发起并发下载
- 同一目标端重复按 `Ctrl+Alt+Shift+V`，只传一次
- 已完成传输的内容再次按热键，只复用 tmp 结果
- 重复收到相同 `file_chunk` / `file_complete` / `file_release` 时，不得破坏最终状态

## 7. 传输与协议要求

### 7.1 文本与图片

- 文本、图片优先走实时广播
- 常规剪贴板消息使用：
  P2P 尝试发送 + STOMP 可靠兜底

### 7.2 文件控制消息

文件控制消息包括：

- `file_request`
- `file_chunk`
- `file_complete`
- `file_error`
- `file_release`

这些消息必须满足：

- 优先定向发给目标 session
- 目标 P2P peer 不存在或未就绪时，回退到 STOMP
- 不能因为“发给了某个别的 peer”就误判成功

## 8. 安全与稳定性要求

当前版本必须具备以下安全边界：

- WebSocket 明确设置单帧读大小限制，不能绕过 HTTP `BodyLimit`
- 管理员修改用户信息时，不存在的用户必须返回 `404`
- 图片复用不能只看“同名同大小”，必须校验内容
- zip 解压不能只信发送端估算值，必须按 zip 实际宣称大小和真实拷贝量双重约束
- zip 解压必须防御 Zip Slip 和符号链接
- Linux Wayland 全局热键关闭不能有 data race
- 暴力破解防护检查逻辑不能有锁检查竞态

## 9. 当前实现备注

### 9.1 Linux Wayland

- 剪贴板链路为正式支持
- 全局热键依赖 `xdg-desktop-portal` GlobalShortcuts
- 如果 portal 不可用，会尝试降级到 X11

### 9.2 Windows

- 进程可以常驻托盘
- 日志窗口可打开查看
- 最小化或关闭日志窗口不应影响后台进程

### 9.3 macOS

- 当前以 GitHub Actions 打包产物为主
- 逻辑要求与 Windows / Linux 桌面端保持一致

### 9.4 无桌面 Linux / SSH / 纯终端模式

这一节描述的是未来需求，不代表当前代码已经支持。

#### 9.4.1 设计目标

无桌面 Linux 模式下，ClipCascade 仍然应该可用，但“可用”的定义要改成：

- 能把共享文本快速带进当前 shell
- 能把图片先落到 tmp，再把图片路径带进当前 shell
- 能把文件传到 tmp，再把结果路径带进当前 shell

而不是继续要求：

- 进入系统图形剪贴板
- 使用桌面全局热键
- 依赖 Explorer / Finder / Dolphin 一类文件管理器目标目录

#### 9.4.2 为什么不能照搬桌面版

无桌面 Linux 里，通常没有统一系统剪贴板 API。

即使某些终端里能用 `Shift+Insert`，那也只是终端自身行为，不代表：

- ClipCascade 可以稳定写入“系统剪贴板”
- 所有 shell / tty / tmux / screen / 物理终端都能统一粘贴

因此，无桌面 Linux 不应把“系统剪贴板”作为强依赖。

#### 9.4.3 推荐交互模型

无桌面 Linux 的主交互模型应改为：

- 命令触发
- shell 绑定触发
- 结果直接插入当前 shell 输入行，或打印到 stdout

推荐命令：

- `cc-paste-text`
- `cc-paste-image`
- `cc-paste-files`

#### 9.4.4 文本需求

文本模式下：

- 用户触发 `cc-paste-text`
- ClipCascade 获取当前共享文本
- 优先目标是把文本直接插入当前 shell 输入行
- 如果当前环境做不到安全插入，则退化为输出到 stdout

这里的“像粘贴一样使用”指的是：

- 用户当前就在 shell 提示符上
- 触发后，文本直接出现在命令行输入框里

而不是要求必须进入图形系统剪贴板。

#### 9.4.5 图片需求

图片模式下：

- 用户触发 `cc-paste-image`
- ClipCascade 把图片物化到 tmp 目录
- 然后把图片文件路径插入当前 shell 输入行
- 如果当前环境不支持安全插入，则打印路径到 stdout

这里插入的是：

- 图片文件路径文本

不是：

- 图片二进制内容本身

这样做的目的，是为了兼容 CLI 工具、脚本、终端里的路径输入场景。

#### 9.4.6 文件需求

文件模式下：

- 用户触发 `cc-paste-files`
- 如果当前共享项只是 `file_stub`，则自动发起或复用真实传输
- 传输完成后，把结果路径插入当前 shell 输入行
- 如果当前环境不支持安全插入，则打印路径到 stdout

路径规则：

- 单文件：插入单文件路径
- 单文件夹：插入单文件夹路径
- 多文件 / 多文件夹 / 混合：插入这次接收结果的根目录路径

#### 9.4.7 触发方式

无桌面 Linux 不应再使用桌面全局热键。

正确触发方式应分两层：

第一层：

- ClipCascade 提供命令

第二层：

- 用户在 `bash` / `zsh` / `fish` 中自行绑定按键到这些命令

例如：

- `Alt-t` -> `cc-paste-text`
- `Alt-i` -> `cc-paste-image`
- `Alt-f` -> `cc-paste-files`

#### 9.4.8 shell 绑定规则

这些按键绑定不应该由 ClipCascade 直接偷偷修改用户配置文件。

原因：

- `bash`、`zsh`、`fish` 的绑定机制不同
- 用户可能使用 `tmux`、`screen` 或自定义终端
- 自动修改用户配置风险高、可预期性差

因此正确要求是：

- ClipCascade 提供建议配置片段或生成器
- 用户自己决定是否写入 shell 配置

可以接受的实现方式：

- 文档给出配置片段
- 或提供例如 `clipcascade-headless init-shell bash|zsh|fish` 的生成命令

但都不应直接静默改用户配置文件。

#### 9.4.9 幂等性要求

无桌面 Linux 模式下也必须满足幂等性：

- 重复触发 `cc-paste-files`，不能重复传输同一份未变化文件
- 已经完成的图片或文件，再次触发时只复用 tmp 结果
- 重复触发文本命令，只重复插入当前共享文本本身，不引入新的副作用

#### 9.4.10 当前状态

截至当前版本：

- 这套无桌面 Linux 模式还没有实现
- 当前代码仍然主要面向桌面 Linux 的 X11 / Wayland 环境
- 因此纯 SSH / 纯 TTY / 机房终端场景，暂时不能宣称“已支持”

## 10. 发布前检查清单

发布前至少确认：

3. `go test -race ./desktop/app ./server/handler` 通过
4. 文档与 README 中的快捷键、粘贴逻辑、tmp 逻辑与代码一致
